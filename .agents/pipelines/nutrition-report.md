# Pipeline: отчет по питанию

Локальный процедурный pipeline для агента питания. Использовать, когда автор передает описание приемов пищи и сырые фото еды, упаковок или этикеток, а агент должен разобрать материалы и создать связанные файлы дневника питания.


## Роль pipeline

Собрать структурированный фактический слой дневника питания:

- отчет по питанию за дату;
- карточки продуктов;
- карточки блюд/рецептов, если в приемах пищи есть приготовленные блюда.

Не анализировать рацион, не считать калории и БЖУ, не давать рекомендации по диете, не делать выводы о стратегии питания. Подсчет калорий должен выполняться отдельно в будущем файле расчета, например `Расчет питания YYYY-MM-DD.md`.

## Входные материалы

- Описание приемов пищи в формате, который автор передаст отдельно. Формат пока не зафиксирован: это может быть текст, таблица, файл, экспорт из другого приложения или другой рабочий формат.
- Отдельные фото еды, готовых блюд, продуктов, упаковок и этикеток - основной источник для разбора приемов пищи, карточек продуктов и карточек блюд. Целевая папка для фото: `Фото/`. На входе фото могут временно находиться в корне vault.

## Выходные файлы

Создавать или обновлять отчет:

```md
Отчет по питанию YYYY-MM-DD.md
```

Не добавлять отдельный H1 с названием файла: в Obsidian имя файла уже работает как заголовок.

Если среди материалов есть продукты, создавать новые или обновлять существующие карточки продуктов в папке `Продукты/`.

Если среди приемов пищи есть приготовленные блюда, создавать новые или обновлять существующие карточки блюд/рецептов в папке `Рецепты/`.

Для карточек продуктов использовать `.agents/skills/product-card.md`.

Для карточек блюд/рецептов использовать `.agents/skills/recipe-card.md`.

Фото продуктов, упаковок и этикеток добавлять в карточки продуктов, а не в отчет.

Фото ингредиентов приготовленного блюда добавлять в карточки соответствующих продуктов, а не в отчет и не в карточку блюда/рецепта.

В карточку блюда/рецепта добавлять только фото готового блюда. Если фото готового блюда нет, не добавлять блок `### Фото` и не ставить `photo: true`.

Перед созданием отчета и карточек переносить входные фото из корня vault в папку `Фото/`. Все ссылки на фото создавать уже после переноса.

## Порядок работы

1. Определить дату отчета по описанию, имени входного файла или явному указанию автора.
2. Нормализовать очевидные опечатки в названиях продуктов и блюд до создания ссылок, имен файлов и карточек.
3. Перенести входные фото из корня vault в папку `Фото/`.
4. Разобрать сырые фото по приемам пищи.
5. Отделить фото полных приемов пищи от фото продуктов, упаковок и этикеток.
6. Сопоставить продукты из описания с видимыми упаковками и этикетками.
7. Создать или обновить карточки продуктов по skill `.agents/skills/product-card.md`.
8. Создать или обновить карточки блюд/рецептов по skill `.agents/skills/recipe-card.md`.
9. Создать или обновить отчет по питанию в корне vault.

Если связи между фото, продуктом, блюдом или приемом пищи недостаточно понятны, не придумывать связь.

## Структура отчета

Структура отчета состоит из трех частей:

1. `## Источники`
2. Блоки приемов пищи в хронологическом порядке.
3. `## Активность и wellness-практики`

```md
## Источники

- Описание питания: [[...]]

## Название приема пищи — время приема не указано

### Продукты

- Продукт — размер/граммовка.

### Фото

<div style="display:grid; grid-template-columns:repeat(4,minmax(0,1fr)); gap:6px; max-width:560px;">
  <img src="Фото/photo-1.jpg" style="width:100%;">
  <img src="Фото/photo-2.jpg" style="width:100%;">
  <img src="Фото/photo-3.jpg" style="width:100%;">
  <img src="Фото/photo-4.jpg" style="width:100%;">
</div>

## Активность и wellness-практики

- Сон: ...
- Активность: ...
- Практики: ...
```

## Правила заполнения

- Блок `## Источники` держать в начале.
- Приемы пищи располагать в хронологическом порядке.
- Каждый прием пищи оформлять отдельным H2-блоком.
- В каждом приеме пищи использовать только подблоки `### Продукты` и `### Фото`, пока автор не утвердит новую структуру.
- Если в тексте есть реальное время приема пищи, использовать его.
- Если время приема пищи неизвестно, писать: `время приема не указано`.
- Внутри приема пищи указывать только продукты/напитки, размеры порций, граммовки и относящиеся фото.
- Если в приеме пищи есть приготовленное блюдо, в отчете указывать только блюдо и съеденную порцию. Ингредиенты блюда не перечислять в отчете; они фиксируются в карточке блюда/рецепта и связанных карточках продуктов.
- Для приготовленного блюда в отчете указывать граммовку съеденной порции. Если граммовка порции не указана, писать `граммовка порции не указана`.
- В карточке блюда/рецепта блок `### Порции` фиксирует порцию как характеристику блюда/рецепта, а не факт употребления в конкретном отчете; если порция не понятна из описания, блок оставлять пустым.
- Очевидные опечатки в названиях продуктов и блюд исправлять при нормализации входных данных; исправленное название использовать в отчете, имени файла и карточках.
- Не придумывать граммовки, состав или время приема пищи.
- Калорийность и БЖУ не добавлять в отчет по питанию; справочные пищевые данные фиксировать только в карточках продуктов с явной пометкой, что они не взяты с упаковки.
- Для продуктов без читаемых данных упаковки или этикетки создавать карточку со справочными данными по skill `.agents/skills/product-card.md`; в карточке обязательно указывать, что данные не взяты с упаковки.
- Если продукт виден на фото, но не указан в описании питания, можно добавить его с пометкой `добавлено по фото`.
- Если размер не известен, писать `размер не указан`.
- Контентные идеи не писать в отчет по питанию.
- Не добавлять таблицы калорий, дневные итоги, оценку дефицита/профицита или рекомендации. Это зона будущего расчета питания.
- Если для продукта или блюда создана карточка, в отчете можно использовать Obsidian-ссылку на нее.
- В блок `### Фото` добавлять только фото полного приема пищи. Фото продуктов, упаковок, этикеток и ингредиентов не добавлять в отчет.

## Фото

Фотографии оформлять HTML-сеткой, потому что:

- Markdown-таблицы ломаются из-за `|250` внутри Obsidian-embed.
- Inline-embeds переносятся непредсказуемо.
- HTML-сетка дает стабильный компактный коллаж.

Правило HTML-сетки:

```md
<div style="display:grid; grid-template-columns:repeat(4,minmax(0,1fr)); gap:6px; max-width:560px;">
  <img src="Фото/photo-file-1.jpg" style="width:100%;">
  <img src="Фото/photo-file-2.jpg" style="width:100%;">
</div>
```

Использовать путь к фото относительно файла отчета. Так как отчет создается в корне vault, для фото из папки `Фото/` писать путь `Фото/photo-file.jpg`.

Если фото для приема пищи не переданы, оставить блок `### Фото` пустым или написать:

```md
- Фото не переданы.
```

Если фото есть, но прием пищи не распознан, добавить в конец отчета блок:

```md
## Фото без привязки

<div style="display:grid; grid-template-columns:repeat(4,minmax(0,1fr)); gap:6px; max-width:560px;">
  <img src="Фото/photo-file.jpg" style="width:100%;">
</div>
```
