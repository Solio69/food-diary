# Конфигурация агентов проекта

Этот vault используется как личная система дневника питания: отчеты по приемам пищи, база продуктов, рецепты, фото еды и будущие расчеты калорий и КБЖУ.

Агент помогает со структурой данных, оформлением Markdown-документов, поддержкой базы знаний и подготовкой фактических отчетов. Агент не является консультантом по питанию, не ставит диагнозы, не дает диетологических рекомендаций и не делает выводы о стратегии питания.

## Рабочая среда

- Корень проекта: `D:\anastasiaStorage\Дневник питания`.
- Пользователь работает с файлами в Obsidian.
- Все служебные файлы и папки для агентов должны начинаться с точки, чтобы быть скрытыми в Obsidian.
- Основной язык документов: русский.
- Предпочтительная кодировка Markdown-файлов: UTF-8.

## Текущая структура

- `.agents/` - служебная папка агентов.
- `.agents/pipelines/` - локальные процедурные pipelines.
- `.agents/pipelines/nutrition-report.md` - pipeline для разбора питания, фото, продуктов, рецептов и отчета.
- `.agents/skills/` - локальные процедурные skills.
- `.agents/skills/product-card.md` - skill для карточек продуктов.
- `.agents/skills/recipe-card.md` - skill для карточек блюд/рецептов.
- `.templates/` - шаблоны пользовательских карточек Obsidian.
- `.templates/Продукт.md` - шаблон карточки продукта.
- `.templates/Блюдо.md` - шаблон карточки блюда/рецепта.
- `.GIT_POLICY.md` - обязательная политика работы AI-агентов с Git.
- `Отчеты/` - пользовательская папка для будущих расчетов и сводок.
- `Продукты/` - база продуктов.
- `Рецепты/` - база рецептов.
- `Фото/` - фото еды, готовых блюд, продуктов, упаковок и этикеток.

## Git-политика

Перед любыми действиями, связанными с Git, агент обязан прочитать и соблюдать `.GIT_POLICY.md`. Этот файл является единственным источником правил Git для проекта.

## Старт работы

- Перед задачей по отчету питания прочитать `.AGENTS.md`, `.agents/pipelines/nutrition-report.md`, `.agents/skills/product-card.md`, `.agents/skills/recipe-card.md`, `.templates/Продукт.md` и `.templates/Блюдо.md`.
- Если папки `Фото/`, `Продукты/` или `Рецепты/` отсутствуют, создать их.
- Для отчета питания искать входное описание в явном указании пользователя, в файле `описание.md` или в другом очевидном текстовом файле с описанием приемов пищи.
- Если дата отчета не указана явно, определять ее по описанию или имени входного файла. Если дата не определяется, спросить пользователя.
- Отчет по питанию создавать в корне vault файлом `Отчет по питанию YYYY-MM-DD.md`.
- Не полагаться на историю чата как на источник данных для отчета; использовать текущие файлы, фото и правила проекта.

## Правила работы с данными питания

- Не придумывать граммовки, состав, время приема пищи или факты о продукте.
- Калорийность и БЖУ в карточках продуктов можно добавлять как справочные данные, если они явно помечены как не взятые с упаковки.
- Если данных не хватает, явно писать `не указано`.
- Разделять фактический слой, расчеты и анализ.
- Фактический слой фиксирует только источники, продукты, блюда, порции и фото.
- Расчет калорий и КБЖУ должен быть отдельным документом или отдельным будущим агентным процессом.

## Pipeline питания

Для задач вида "составить отчет по питанию", "разобрать фото питания", "пополнить продукты" или "создать рецепты по приемам пищи" использовать `.agents/pipelines/nutrition-report.md`.

Ключевые правила pipeline:

- На входе используются описание приемов пищи и сырые фото еды, готовых блюд, продуктов, упаковок или этикеток.
- На выходе создаются или обновляются отчет по питанию, карточки продуктов и карточки блюд/рецептов.
- Для карточек продуктов использовать `.agents/skills/product-card.md`.
- Для карточек блюд/рецептов использовать `.agents/skills/recipe-card.md`.
- Создавать или обновлять файл `Отчет по питанию YYYY-MM-DD.md`.
- Новые продукты фиксировать в `Продукты/`, существующие карточки продуктов обновлять фактическими данными из текущих материалов.
- Новые приготовленные блюда фиксировать в `Рецепты/`, существующие карточки блюд/рецептов обновлять фактическими данными из текущих материалов.
- Карточки продуктов создавать по шаблону из `.templates/Продукт.md` и сохранять его структуру при обновлении.
- Карточки блюд/рецептов создавать по шаблону из `.templates/Блюдо.md` и сохранять его структуру при обновлении.
- YAML-поле `photo` в карточках является чекбоксом наличия фото; если к карточке добавлено подходящее фото, агент ставит `photo: true`, а сами изображения крепит ниже через Obsidian image-embeds `![[Фото/...|260]]`.
- YAML-поле `done` является пользовательским чекбоксом проверки; агенты никогда не меняют `done`.
- Если пользователь подтверждает карточку со справочными данными через `done`, карточка считается валидной для базы.
- Перед составлением отчета и карточек переносить входные фото из корня vault в `Фото/` и создавать ссылки уже на новое расположение.
- На этапе разбора нормализовать очевидные опечатки в названиях продуктов и блюд до создания ссылок, имен файлов и карточек.
- В карточках продуктов не использовать блок `Источники`; блоки оформлять H3-заголовками.
- После `### Описание` использовать ровно один блок пищевых данных: `### Данные с упаковки`, если данные с упаковки читаются на фото или явно указаны автором; `### Справочные данные`, если данных с упаковки нет или они не читаются.
- В блоке пищевых данных обязательно фиксировать калорийность и белки на 100 г/100 мл; жиры и углеводы фиксировать, если есть данные.
- Если данные с упаковки читаются частично и в них нет обязательной калорийности или белков, недостающее обязательное значение можно добавить в `### Данные с упаковки` как справочное с явной пометкой `справочно, не с упаковки`.
- Полный порядок блоков карточки продукта: `### Описание`, затем `### Данные с упаковки` или `### Справочные данные`, затем `### Фото`, `### Использовано в блюдах`, `### Использование в питании`.
- Блюдо определять по описанию автора, а не только по фото.
- Если автор упоминает блюдо или порцию, считать, что еда готовилась как блюдо; состав фиксировать в карточке блюда/рецепта, а употребленную долю фиксировать в отчете.
- В карточке блюда/рецепта блок `### Порции` фиксирует порцию как характеристику блюда/рецепта, а не факт употребления в конкретном отчете; если порция не понятна из описания, блок оставлять пустым.
- В карточках блюд/рецептов не использовать блок `Источники`; использование в отчете фиксировать нижним блоком `### Использование в отчетах`.
- Карточки блюд/рецептов оформлять H3-заголовками в порядке: `### Описание`, `### Состав блюда`, `### Порции`, опционально `### Фото`, `### Использование в отчетах`.
- В карточке блюда/рецепта прикреплять только фото готового блюда. Фото продуктов, упаковок, этикеток и ингредиентов фиксировать в карточках продуктов.
- Если фото готового блюда нет, не добавлять в карточку блюда/рецепта блок `### Фото` и не ставить `photo: true`.
- В отчете по питанию приготовленное блюдо указывать одной строкой с употребленной порцией; ингредиенты блюда фиксировать в карточке блюда/рецепта и связанных карточках продуктов, а не в отчете.
- Если для блюда понятна доля или порция, указывать ее в отчете. Если граммовка употребленной порции блюда не указана, добавлять `граммовка порции не указана`.
- Если непонятно, какие данные относятся к составу блюда, остановиться и спросить автора.
- Не добавлять H1 с названием файла.
- Структура отчета: `## Источники`, затем H2-блоки приемов пищи в хронологическом порядке.
- Внутри приема пищи использовать только `### Продукты` и `### Фото`, пока пользователь не утвердит другую структуру.
- В отчете оставлять только фото полных приемов пищи; фото продуктов, упаковок, этикеток и ингредиентов переносить в карточки продуктов. В карточки блюд/рецептов добавлять только фото готового блюда.
- Фото в отчете оформлять через Obsidian image-embeds `![[Фото/...|260]]`.
- Целевая папка для фото: `Фото/`; на входе фото могут временно лежать в корне vault.
- Если фото не переданы, оставлять блок фото пустым или писать `- Фото не переданы.`
- Если фото есть, но прием пищи не распознан, добавлять `## Фото без привязки`.

## Стиль файлов

- Писать документы коротко, структурно и пригодно для Obsidian.
- Использовать wiki-ссылки Obsidian там, где это удобно для связей между продуктами, рецептами, отчетами и источниками.
- Не добавлять служебный агентный шум в пользовательские документы.
- Не создавать видимые служебные файлы без прямой необходимости.
- В описаниях процессов, конфигурациях и skills фиксировать только актуальное состояние работы. Не описывать прежние варианты, переходы между версиями и историю изменений; неактуальную информацию удалять или заменять актуальной.
- Сохранять пользовательские правки и не откатывать изменения, сделанные не агентом.
